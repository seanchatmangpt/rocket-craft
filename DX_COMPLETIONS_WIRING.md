# Rocket CLI DX: Completions, Man Pages & Rich Help — Wiring Guide

This document explains how to wire the new `tools/rocket-sdk/src/completions.rs`
module into the `rocket` CLI. All Rust logic lives in that module; the snippets
below are the only edits needed to the (owner-controlled) files
`tools/rocket-sdk/src/lib.rs` and `tools/rocket-cmd/src/main.rs`.

The module is **std + serde only** — no new dependencies. `serde`/`serde_json`
are already in `rocket-sdk/Cargo.toml`, so **no `Cargo.toml` changes are
required**.

---

## 1. `tools/rocket-sdk/src/lib.rs`

Add one line alongside the other `pub mod` declarations at the top:

```rust
pub mod completions;
```

That's the entire library-side change.

---

## 2. New Cargo dependencies

**None.** The module uses only `std` and the already-present `serde` /
`serde_json`. The completion scripts are generated by hand-rolled string
templates (a declarative `CommandSpec` is the single source of truth), so it does
**not** pull in `clap_complete` — keeping `rocket-sdk` free of any clap coupling.

> Note: `rocket-cmd` already depends on `clap_complete` and uses it for the
> existing `completions` subcommand. You can keep that, or switch the subcommand
> to the new generator (snippet 3a) to get a single, richer source of truth that
> also drives man pages and help. The new generator additionally documents the
> `man` subcommand and emits value hints (platforms, shells).

---

## 3. `tools/rocket-cmd/src/main.rs`

### 3a. (Optional) Replace the `Completions` handler with the SDK generator

The existing `Commands::Completions { shell }` arm uses `clap_complete`. To route
it through the new SDK generator (so completions, man pages, and help all share
one spec), change the handler body. Keep the `clap_complete::Shell` arg type for
CLI parsing, or swap to a plain `String` and parse with
`rocket_sdk::completions::Shell::parse`.

Add to imports:

```rust
use rocket_sdk::completions::{
    self, generate_completions, generate_man_page, generate_subcommand_man_page,
    install_instructions, render_help, rocket_command_spec, Shell as RocketShell,
};
```

Replace the `Commands::Completions { shell }` match arm with:

```rust
Commands::Completions { shell } => {
    // Map clap_complete::Shell -> rocket_sdk::completions::Shell.
    let sdk_shell = match shell {
        Shell::Bash => RocketShell::Bash,
        Shell::Zsh => RocketShell::Zsh,
        Shell::Fish => RocketShell::Fish,
        Shell::PowerShell => RocketShell::PowerShell,
        // clap_complete::Shell is non_exhaustive; default unknowns to bash.
        _ => RocketShell::Bash,
    };
    let spec = rocket_command_spec();
    print!("{}", generate_completions(sdk_shell, &spec));
    // Print install hint to stderr so it doesn't pollute the script on stdout.
    eprintln!("\n# To install:\n{}", install_instructions(sdk_shell));
}
```

> If you prefer not to depend on `clap_complete` at all, change the field to
> `shell: String` in the `Commands::Completions` variant and parse with
> `RocketShell::parse(&shell).ok_or_else(|| eyre!("unknown shell: {shell}"))?`.

### 3b. Add a `man` subcommand

Add a variant to the `Commands` enum:

```rust
/// Generate troff man pages for rocket (all, or a single subcommand)
Man {
    /// Subcommand to document; omit for the top-level page
    command: Option<String>,
},
```

Add the match arm in `main()`:

```rust
Commands::Man { command } => {
    let spec = rocket_command_spec();
    match command {
        Some(name) => match spec.find_sub(&name) {
            Some(sub) => print!("{}", generate_subcommand_man_page(&spec, sub)),
            None => {
                eprintln!("Unknown command: {name}");
                if let Some(s) = completions::did_you_mean(&name, &spec.subcommand_names()) {
                    eprintln!("Did you mean '{s}'?");
                }
            }
        },
        None => print!("{}", generate_man_page(&spec)),
    }
}
```

Usage:

```bash
rocket man                       # full troff man page on stdout
rocket man > rocket.1            # save it
rocket man build > rocket-build.1
man ./rocket.1                   # preview locally
```

### 3c. Improved `--help` + "Did you mean?" for unknown subcommands

clap exits before `main()`'s match runs when it can't parse args, so to layer in
the rich help and fuzzy suggestions, intercept parsing. Replace
`let cli = Cli::parse();` with a `try_parse` that falls back to the rich
renderer:

```rust
let cli = match Cli::try_parse() {
    Ok(cli) => cli,
    Err(e) => {
        use clap::error::ErrorKind;
        let spec = rocket_command_spec();
        let color = std::env::var_os("NO_COLOR").is_none();
        match e.kind() {
            // Top-level `--help` / no args: show the rich grouped help.
            ErrorKind::DisplayHelp | ErrorKind::MissingSubcommand => {
                print!("{}", render_help(&spec, None, color));
                std::process::exit(0);
            }
            // Unknown subcommand: suggest the closest match.
            ErrorKind::InvalidSubcommand | ErrorKind::UnknownArgument => {
                // Best-effort: recover the offending token from argv.
                if let Some(arg) = std::env::args().nth(1) {
                    print!("{}", render_help(&spec, Some(&arg), color));
                } else {
                    print!("{}", render_help(&spec, None, color));
                }
                std::process::exit(2);
            }
            _ => e.exit(),
        }
    }
};
```

> `render_help(&spec, Some("biuld"), color)` prints
> `Unknown command: 'biuld'` followed by `Did you mean 'build'?` and a pointer to
> `rocket --help`. `render_help(&spec, Some("build"), color)` prints full,
> example-rich help for the `build` subcommand — so you can also wire a
> `rocket help <cmd>` alias to it.

## What changed & why it's 1000x

| Capability | Before | After |
|---|---|---|
| **Shell completions** | `clap_complete` default scripts only; no value hints; bash/zsh/fish/pwsh but undocumented | A declarative `CommandSpec` source of truth generates bash/zsh/fish/PowerShell scripts with subcommand descriptions, nested subcommands (`crypto generate`, `pwa lint`), and value hints (platforms, shells) — all testable as plain strings. |
| **Man pages** | None | `generate_man_page` (full troff `.1`) + per-subcommand pages with NAME/SYNOPSIS/DESCRIPTION/OPTIONS/EXAMPLES sections. `rocket man > rocket.1` → real `man` integration. |
| **Help** | clap's flat auto-help | `render_help` with **grouped** commands (lifecycle / quality / assets / info / DX), a **"Common workflows"** section (first-time setup, build & ship, PWA, Android), per-subcommand example-rich help, and ANSI color gated behind a bool (honors `NO_COLOR`). |
| **Typo recovery** | clap's terse "unrecognized subcommand" | `did_you_mean` (Levenshtein + prefix bias) → `rocket biuld` suggests `build`, `rocket doctr` → `doctor`. |
| **Install UX** | Manual guesswork | `install_instructions(shell)` prints exact paths/commands per shell. |
| **Dependencies** | — | **Zero new deps.** Self-contained, serde-serializable spec reusable by docs/build tooling. |
| **Tests** | — | 21 `#[cfg(test)]` tests: all subcommands present, all four shells non-empty & containing expected tokens, man-page sections, Levenshtein correctness, did-you-mean hits/misses/prefix, colorized vs plain help, install paths, JSON round-trip. |

The leverage: a **single declarative spec** (`rocket_command_spec()`) now drives
completions, man pages, and help simultaneously. Add a flag once and every
surface updates — instead of editing four hand-maintained completion scripts plus
docs by hand.
