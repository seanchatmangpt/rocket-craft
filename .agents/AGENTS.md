# Project-Scoped Agent Rules

## The Combinatorial Maximalist Doctrine

These are the absolute architectural laws governing the generation of game servers and client projections. They apply to all agents operating within this workspace.

### 1. Agent Jidoka (The Autonomation Law)
- **Halt on Paradox:** If a generated artifact fails SHACL validation, fails to compile, or contains a syntactic paradox, the agent MUST immediately halt the pipeline and flag a failure.
- **No Mock Laundering:** Never substitute a fake runtime, a Python mock script, a stubbed string test, or placeholder code to bypass a failing test. A passing test is only valid if it runs against the native target engine compiler.
- **Wrong Work is Discovery:** Accept failures as valuable discovery, but never admit them as authoritative standing.

### 2. Branchless Typestates ($A = \mu(O^*)$)
- **Eliminate Runtime Branching:** Structural and mathematical constraints must be pushed into the compiler type system (e.g., Rust Zero-Sized Types, C++ Concepts, Const Generics).
- **Compile-Time Rejection:** If an object state violates the ontology (e.g., adding Mass to Velocity), it must physically fail to compile. Do NOT use `if (state == "invalid")` checks at runtime.
- **The Graph is the Code:** Tera templates must stamp out code exactly as projected by the `SPARQL` subset, guaranteeing exhaustive combinatorial closure.
- **SIMD/Scalar Equivalence:** Generated branchless authority kernels must prove identical deterministic output across both SIMDe/WASM vector paths and scalar fallbacks for the exact same byte-class inputs.
- **Performance Expectation:** SIMDe is the portable vector vocabulary for ggen-manufactured semantic law. It provides ~500x throughput over UE4 Blueprint logic by bypassing the VM, UObjects, and branchy logic in favor of contiguous byte arrays. Scalar and SIMD paths must be receipt-equivalent.

### 3. The Three-Resolution Networking Model
- **Byte-Class Authority:** The server owns standing, not visual detail. Authoritative state must be compacted into discrete byte-class vectors (e.g., `damage_class: 1 byte`) and Merkle-rooted assembly graphs.
- **Peer Projection:** Clients may exchange high-resolution visual details peer-to-peer (sparks, particle effects, animations) but these hold no authoritative standing.
- **Client as Projector:** The local client renders the absolute maximum fidelity, but relies entirely on the server for the admitted truth. *The server does not synchronize the object; it synchronizes the right to believe the object.*

### 4. SPARQL Subset Bounding (Anti-Cartesian Exhaustion)
- **Never Export the Whole Graph:** Do not attempt to load the entire multi-gigabyte ontology into the compiler or the WASM client.
- **Bounded Selection:** Use strict, target-oriented `SPARQL SELECT` queries to extract only the exact deterministic subset needed for a specific code generation rule.
- **Strict Determinism:** All `SPARQL` extraction queries must use `ORDER BY` to guarantee perfectly repeatable parallel generation paths.

### 5. Playwright Manufacturing & Cryptographic Standing
- **Admittance Requires Actuation:** A generated artifact is not considered "Done" just because it compiles. 
- **Visual Delta Proof:** The final acceptance gate requires the artifact to load in its native runtime (e.g., UE4 HTML5 via Playwright), actuate, and demonstrate an observable visual delta.
- **BLAKE3 Receipts:** The final state must be hashed into an unforgeable BLAKE3 receipt, mathematically proving the compilation and the execution.

# AGENTS.md — TAI Lifecycle Discipline Override

## Purpose

This project does not optimize for code production, feature count, commit count, ontology size, document size, benchmark scores, or apparent progress.

This project optimizes for manufacturable, supportable, verifiable systems that can survive independent inspection.

The standard is not "AI generated."

The standard is concept exploration through system disposal.

The agent must operate as if it is responsible for the system for its entire lifecycle.

---

# Fundamental Rule

Never confuse activity with progress.

Never confuse progress with standing.

Never confuse standing with proof.

Every claim must identify:

* Observation
* Evidence
* Scope
* Assumptions
* Residual risks
* Required verification

If any of these are missing, the claim is incomplete.

---

# TAI Quality Leadership Rule

The customer mission is the primary optimization target.

Not:

* code volume
* ontology volume
* generated artifacts
* benchmarks
* demonstrations

The question is always:

"What mission capability was improved?"

If the answer is unclear, stop and reassess.

# Receipt Rule

# TAI Status Reporting Format

All progress and status reports MUST strictly adhere to this format. Collapse all tactical progress into exact, uninflated categories. Do not conflate build logistics with mission capability.

**Status:** [BLOCKED / PARTIAL / PARTIAL_ALIVE candidate / ALIVE_UNDER_SCOPE]
**Object under test:** [Exact pipeline segment or artifact being worked on, e.g., UE4 HTML5 packaging. Do not use strategic labels like "Eden world" for tactical tasks.]
**Observed evidence:** [Exact file path, command, exit code, or artifact hash. No adjectives.]
**Failure:** [Specific blocker, e.g., missing PhysXVehicle_O3.a or Python mismatch.]
**Repair:** [Specific action taken, e.g., ran HTML5Setup.sh or replaced shell script with Rust runner.]
**Receipt required:** [The physical evidence needed to prove the next gate, e.g., package exists, browser launches, visual delta captured, replay script generated.]
**Residuals:** [What remains unproven, e.g., no gameplay proof, no walkthrough proof, no ggen-world proof yet.]

*Build logistics progress is not runtime proof.*
*Runtime proof is not walkthrough proof.*
*Walkthrough proof is not ALIVE unless independently replayed and receipted.*

# The Projection Law

**We own the authority. Unreal owns the pixels.**

| Layer                          | Owns                                                                                        | Does not own                            |
| ------------------------------ | ------------------------------------------------------------------------------------------- | --------------------------------------- |
| **RDF / ggen**                 | semantic authority, topology, typestates, byte classes, Render BOM, walkthrough coordinates | meshes, shaders, lighting, WebGL assets |
| **Generated C++ / DataTables** | static bridge from admitted graph into UE4 runtime                                          | visual authorship                       |
| **UE4**                        | models, materials, lighting, physics projection, packaging, WebGL/WASM output               | semantic standing                       |
| **Playwright / verifier**      | browser proof, walkthrough proof, receipt capture                                           | authority invention                     |

**ggen emits the law. UE4 emits the pixels.**

Do not generate WebGL assets from ontology. Unreal generates and packages render assets. `ggen` generates semantic authority artifacts consumed by Unreal C++: headers, enums, structs, constants, DataTables, Render BOM metadata, walkthrough coordinates, byte-class matrices, and receipt paths.

`VaRest` is removed only when UE4 no longer requires runtime REST/JSON/Blueprint plugin logic to obtain world structure or Semantic LOD state.
