## Key Concepts

### The Engine (anti-llm-cheat-lsp)
```
Source Code
    ↓ [scan_file/scan_directory]
Observations (raw patterns)
    ↓ [evaluate_diagnostics]
AntiLlmDiagnostic (with blocking: bool)
    ↓ [bridge converters]
    → JSON (MCP, CLI)
    → LSP Diagnostic (editor)
    → Refusal (admission gate)
```

**Key Insight**: All 4 integrations feed on the same engine output; they just transform it differently.

### Type Conversions
| From | To | Module | When |
|------|----|---------|----|
| Observation | JSON | serde | MCP tools, CLI |
| AntiLlmDiagnostic | LSP Diagnostic | unify-lsp gate | Editor display |
| AntiLlmDiagnostic | Refusal | unify-admission gate | Commit enforcement |

### Data Sources
- **Observations**: Generated by 14 parsers (Rust, TypeScript, Markdown, JSON, etc.)
- **Diagnostics**: Generated by 17 rule engines (surface, authority, receipts, etc.)
- **Config**: `AntiLlmConfig` (currently hardcoded, future: load from `anti-llm.toml`)

## Risk & Mitigation

| Risk | Mitigation |
|------|-----------|
| Type conversion bugs | Comprehensive unit tests + manual validation |
| LSP integration complexity | Early prototyping; separate from other phases |
| Performance on large repos | Benchmarking; caching for future |

**Overall**: Low to medium risk; each phase independently rollbackable
