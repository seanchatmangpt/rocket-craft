## 1. Purpose and Scope

This policy establishes [ORGANIZATION]'s risk management framework as it applies to the Rocket Craft monorepo, its supply chain of Rust crates, npm packages, and third-party services (Supabase, Android SDK). It supports SOC2 Type II CC9.1 and CC9.2 by ensuring that risks are identified, assessed, mitigated, and continuously monitored.

---

## 2. Risk Identification (CC9.1)

2.1 Risk identification occurs through three continuous mechanisms:

   a. **Automated dependency scanning** — `cargo audit` (via `cargo make audit` in `tools/`) scans all Rust workspaces against the RustSec advisory database. `npm audit` covers `pwa-staff/` node dependencies.
   b. **CI pipeline signals** — failed builds, test regressions, and secret scans (`.github/workflows/ci.yml`) surface risks on every push.
   c. **Quarterly risk review** — the Engineering Lead and Security Auditor jointly review open vulnerabilities, architectural law violations from `./rocket audit`, and third-party service security bulletins.

2.2 Identified risks are recorded in the risk register maintained in the internal ticketing system with the label `soc2-risk`.

---

## 3. Software Bill of Materials (SBOM) (CC9.1)

3.1 [ORGANIZATION] maintains a current SBOM for all Rust workspaces. The SBOM is generated by running `cargo sbom --format cyclonedx-json` from each workspace root and stored in the `sbom/` directory at the repository root.

3.2 SBOM generation is incorporated into the quarterly risk review. Any new direct dependency added to a workspace `Cargo.toml` requires approval from the LeadDeveloper and a note in the pull request describing the dependency's purpose, licence, and last audit date.

3.3 The `tools/` workspace uses `resolver = "3"` (Rust 2024 edition). All other workspaces use `resolver = "2"`. Mixing resolvers within a workspace is prohibited to prevent unexpected feature unification.

3.4 The `pwa-staff/` TypeScript PWA SBOM is generated via `npm sbom --sbom-format cyclonedx` and updated quarterly alongside the Rust SBOMs.

## 4. Dependency Vulnerability Review (CC9.1)

4.1 **Critical/High CVEs**: Must be addressed within **24 hours** of detection. If a patched version is unavailable, the dependency must be removed or isolated behind a feature flag. A GitHub issue is opened immediately with the `severity:critical` label.

4.2 **Medium CVEs**: Must be patched within **7 days**. If a patch is not available, a documented compensating control must be in place.

4.3 **Low CVEs**: Addressed in the next regular maintenance cycle, no later than the next quarterly review.

4.4 Known false positives or accepted risks must be documented in `.cargo/audit.toml` (for Rust) or a local `audit-exceptions.json` file (for npm), both committed to the repository and referenced in the risk register.

4.5 The external path dependencies `wasm4pm-compat` (at `/Users/sac/wasm4pm-compat`) and `clap-noun-verb` (at `../../../clap-noun-verb`) in `tools/rocket-cmd/Cargo.toml` are a known supply chain risk. They must be replaced with vendored or crates.io versions before any production deployment of `rocket-cmd`. This is tracked in the risk register as `RISK-001`.

## 5. Third-Party Vendor Assessment (CC9.2)

### 5.1 Supabase

Supabase provides the authentication and database backend for the `pwa-staff` PWA. [ORGANIZATION] assesses Supabase annually against the following criteria:

- SOC2 Type II report availability (required).
- Data residency compliance with applicable regulations.
- Encryption at rest and in transit (AES-256 / TLS 1.2+).
- Incident notification commitments (72-hour breach notification).
- API stability and deprecation notice period (minimum 6 months).

The Supabase local instance for development is run via `supabase start` (requires Docker). The local anon key (`sb_publishable_ACJWlzQHlZjBrEguHvfOxg_3BJgxAaH`) is safe to commit as it only functions against localhost. Production keys are stored in the secrets vault and never committed.

### 5.2 GitHub Actions

GitHub Actions is the CI/CD execution environment. It is assessed annually for:

- SOC2 Type II / ISO 27001 certification status.
- Secret management practices (repository secrets, environment protection rules).
- Runner isolation (use of GitHub-hosted runners; custom runners require security review).

### 5.3 Android SDK and Google Play

Android build toolchain dependencies (Android SDK, NDK, Java) are pinned to specific versions documented in `.github/workflows/ci.yml`. Changes to SDK versions require DevOps review and testing against all Android-targeted UE4 projects (`SurvivalGame`, `Brm`, `InfinityBlade4`).

## 6. Business Continuity (CC9.1)

6.1 The `project-manifest.json` at the repository root is the single authoritative registry of all UE4 projects, targets, and platforms. It must be backed up as part of any repository backup procedure.

6.2 Unreal Engine binary assets (`.uasset`, `.umap` files in `versions/`) must not be merged as text diffs. They are managed with Unreal's built-in source control or copied wholesale. Backup of UE4 binary assets is the responsibility of the platform engineering team using dedicated binary asset storage (e.g., Perforce, Git LFS with versioned storage).

6.3 The `asset-pipeline` workspace (in `.claude/worktrees/`) provides autonomous 3D-model-to-UE4-FBX conversion. Business continuity requires that `pipeline.toml` and `pipeline-manifest.json` run logs are backed up alongside source.

6.4 Recovery Time Objective (RTO) for the `pwa-staff` PWA: **4 hours** (re-deploy from `dist/` artifact or rebuild from source). Recovery Point Objective (RPO): **1 hour** (Supabase point-in-time recovery).

## 7. Risk Reporting

7.1 The risk register is reviewed quarterly. A summary of open risks, mitigations applied, and newly accepted risks is included in the quarterly security review package submitted to [ORGANIZATION] leadership.

7.2 Any risk rated Critical that remains open for more than 48 hours must be escalated to the Head of Engineering with a written justification.

---

## 8. Technical Enforcement

This policy is technically enforced via:

- `cargo make audit` in `tools/` — automated RustSec vulnerability scanning
- `./rocket audit` — architectural law compliance via WASM plugins (`knhk`)
- `tools/rocket-sdk/src/access_control.rs` — policy engine governs access to high-risk resources
- `python3 check-dependencies.py` — verifies UE4 plugin presence before builds
