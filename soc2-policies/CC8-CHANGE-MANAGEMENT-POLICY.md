## 1. Purpose and Scope

This policy defines how changes to the Rocket Craft monorepo — including Rust workspace code, Unreal Engine project assets, TypeScript PWA source, CI pipeline definitions, and access control policy files — are proposed, reviewed, approved, deployed, and rolled back. It applies to all contributors to the repository and supports SOC2 Type II CC8.1.

---

## 2. Branch and Repository Protection (CC8.1)

2.1 The `main` and `master` branches are protected. Direct pushes are prohibited; all changes must arrive via pull request.

2.2 Pull requests targeting `main` require a minimum of **two (2) approving reviews**: one from a peer Developer or LeadDeveloper, and one from a LeadDeveloper or above.

2.3 Branch names for all work follow the convention `<type>/<descriptor>-<id>` (e.g., `feat/access-control-cc6`, `fix/audit-chain-mismatch`). Deviations must be approved by the Engineering Lead.

2.4 Force-pushes to `main` and `master` are unconditionally prohibited. The `./rocket` CLI enforces this at the wrapper level; CI rejects force-pushed refs.

---

## 3. Pull Request Requirements (CC8.1)

3.1 Every pull request must include:

   - A description of the change and its business or security rationale.
   - A reference to the associated ticket or issue.
   - Confirmation that `./rocket test` passes locally.
   - For changes to access control policy files (`soc2-policies/`, `access_control.rs`): an updated permission matrix generated by `rocket policy matrix`.

3.2 Pull requests touching `tools/rocket-sdk/src/access_control.rs` or any `soc2-policies/` file additionally require review by the SecurityAuditor role.

3.3 Changes to `.github/workflows/ci.yml`, `project-manifest.json`, or Android keystores require DevOps review plus a formal change request ticket.

---

## 4. CI Gate Requirements (CC8.1)

4.1 The CI pipeline (`.github/workflows/ci.yml`) must pass before any pull request can be merged. The pipeline enforces:

   - **`pwa-staff` job**: `npm ci` → `npm run lint` → `npx tsc --noEmit` → `vitest run` → `npm run build`.
   - **`chicago-tdd-tools` job**: `cargo build --all-features` → `cargo test --all-features`.
   - **Secret scan**: GitHub secret scanning must report zero new findings.
   - **Semantic audit**: `./rocket audit` must produce a passing affidavit receipt with no law violations.

4.2 A failed CI gate blocks merge regardless of approvals. No bypass mechanism is provided.

4.3 Rust workspace changes (`nexus-engine/`, `blueprint-rs/`, `unify-rs/`, `infinity-blade-4/mud/`) not yet covered by CI must be tested locally and evidenced by a passing `cargo test --all` output attached to the pull request as a comment before merge.

---

## 5. Semantic Law Compliance (CC8.1)

5.1 `./rocket audit` loads WASM compliance plugins from the `capabilities/` directory via the `knhk` crate. Laws enforce architectural invariants such as "no circular crate dependencies" and "all Android-targeted UE4 projects must have a keystore."

5.2 Any new architectural law must be encoded as a WASM plugin and committed to `capabilities/` alongside the change that introduces the corresponding constraint.

5.3 The `./rocket wasm --file path/to/plugin.wasm` command executes a single WASM law plugin for development-time validation before CI.

---

## 6. Deploy Approval (CC8.1)

6.1 Production deployments (PWA to hosting, Supabase schema migrations, Android keystore rotation) require:

   a. A merged pull request (per sections 2–4 above).
   b. Explicit approval from the DevOps Engineer via the deployment ticket.
   c. For Supabase schema changes: a second approval from the LeadDeveloper and a backup of the existing schema.

6.2 `EnvironmentIsolation::check_isolation("production", ...)` warnings must be resolved before a production deployment proceeds. The policy engine surfaces these warnings via `rocket doctor` when `ROCKET_ENV=production`.

6.3 Android keystore deployments are performed exclusively using `./rocket crypto generate` with the `--env production` flag. Keystores are stored outside the repository and referenced via environment variable.

---

## 7. Rollback Procedures (CC8.1)

7.1 Every deployment must have a documented rollback plan included in the deployment ticket before deploy begins.

7.2 Application rollbacks use `git revert <merge-commit>` to create a new revert commit, which then follows the standard PR process on an accelerated timeline (single reviewer, 30-minute review window).

7.3 Supabase schema rollbacks use down-migration scripts committed alongside every up-migration. The DevOps engineer executes the down-migration via Supabase CLI.

7.4 In the event of an emergency rollback, the on-call DevOps engineer may execute the rollback without a full PR review but must open a retrospective ticket within 24 hours documenting the incident (see the Incident Response Policy).

---

## 8. Change Records and Audit Trail (CC8.1)

8.1 All merged pull requests constitute the primary change record. Pull request numbers and merge commit hashes are referenced in the BLAKE3 audit chain via `./rocket audit` event IDs.

8.2 `python3 validate-assets.py` must pass before any commit is created; it blocks references to `Highrise` or `Brm-HTML5-Shipping` asset paths. This check is enforced as a pre-commit hook.

8.3 `python3 check-dependencies.py` must be run with `UE4_ROOT` set for any change that touches UE4 plugin configuration, confirming that `WebSocketNetworking` and `VaRest` plugins are present.

---

## 9. Technical Enforcement

This policy is technically enforced via:

- `.github/workflows/ci.yml` — automated gate for every pull request
- `./rocket audit` — semantic law compliance via WASM plugins
- `./rocket test` — local test runner delegating to per-workspace `cargo test`
- `python3 validate-assets.py` — pre-commit asset reference guard
- `tools/rocket-sdk/src/access_control.rs` `PolicyEngine` — enforces who may trigger `CiPipeline` and write `Manifest`
